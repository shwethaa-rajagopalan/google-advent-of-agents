# sciontool Agent-to-Hub Authentication

## Status
**Implemented** (2026-01-31)

## 1. Overview

This document specifies the authentication and authorization mechanism for `sciontool` when communicating with the Scion Hub API from within an agent's container. This ensures that agents can securely report their status and perform authorized actions without requiring user credentials.

### Goals

1. **Secure Identity** - Establish a cryptographically verifiable identity for each agent instance.
2. **Least Privilege** - Use scoped permissions to limit what an agent can do on the Hub.
3. **Automated Provisioning** - Tokens should be provided automatically during agent startup.
4. **Independent Lifecycle** - Agent tokens are tied to the agent's lifecycle, not the user's session.

---

## 2. Authentication Mechanism

### 2.1 Hub-Generated JWT

When the Hub provisions an agent (or instructs a Runtime Broker to do so), it generates a short-lived (or session-bound) JWT.

- **Issuer:** Scion Hub
- **Signing Algorithm:** RS256 (Asymmetric) or HS256 (Symmetric, simpler for initial implementation).
- **Private Key:** Configured on the Hub server.

### 2.2 Token Claims

The JWT includes signed claims that define the agent's identity and its authorization boundaries.

```json
{
  "iss": "scion-hub",
  "sub": "agent_uuid_12345",
  "aud": "scion-hub-api",
  "iat": 1738243200,
  "exp": 1738329600,
  "grove_id": "grove_uuid_67890",
  "scopes": [
    "agent:status:update"
  ]
}
```

| Claim | Description |
|-------|-------------|
| `sub` | The unique ID of the agent instance. |
| `grove_id` | The ID of the grove the agent belongs to. |
| `scopes` | A list of strings representing authorized actions. |

---

## 3. Token Distribution

The token is passed to the agent container via environment variables during the `provision` or `start` phase of the agent lifecycle.

### 3.1 Environment Variables

The Runtime Broker ensures the following environment variable is set in the agent's container:

- `SCION_HUB_TOKEN`: The JWT generated by the Hub.
- `SCION_HUB_URL`: The endpoint for the Hub API.

### 3.2 `sciontool` Usage

`sciontool` will automatically read `SCION_HUB_TOKEN` and include it in the `Authorization` header for all requests to the Hub.

```http
POST /api/v1/agents/agent_uuid_12345/status HTTP/1.1
Host: hub.example.com
Authorization: Bearer <JWT>
Content-Type: application/json

{
  "status": "running",
  "message": "Initializing workspace..."
}
```

---

## 4. Authorization Scopes

Authorization is enforced by the Hub API based on the `scopes` claim in the JWT.

### 4.1 Initial Scopes

Initially, agents are granted the minimum permission required to report their own progress:

- **`agent:status:update`**: Allows the agent to call `POST /api/v1/agents/{self_id}/status` to update its own status and heartbeat.

### 4.2 Future Scopes

As the platform evolves, specific agents may be granted broader capabilities:

- **`grove:agent:create`**: Allows an agent to provision "sub-agents" or peer agents within the same grove.
- **`grove:secret:read`**: Allows an agent to retrieve specific grove-scoped secrets (e.g., API keys for external services).
- **`agent:log:append`**: Explicit scope for streaming logs back to the Hub (if not handled by the Runtime Broker).

---

## 5. Security Considerations

### 5.1 Token Expiry

Tokens should have an expiration (`exp`) matching the expected duration of the agent's task, plus a reasonable buffer. If an agent runs longer than expected, it may need a mechanism to refresh its token, though initially, we will use fixed-duration tokens.

### 5.2 Scope Validation

The Hub API MUST verify:
1. The JWT signature is valid.
2. The `sub` claim matches the agent ID in the request path (for `agent:*` scopes).
3. The `grove_id` matches the resource's grove (for `grove:*` scopes).

### 5.3 Private Key Management

The Hub's private key used for signing these tokens must be protected. In a distributed "Hosted" architecture, only the Hub needs the private key; Runtime Brokers and `sciontool` only receive the signed result.

## 6. Implementation Plan

1. **Hub Logic**: Implement a `GenerateAgentToken(agentID, groveID)` function in `pkg/hub`.
2. **Provisioning**: Update the agent provisioning flow to call this function and pass the token to the `RuntimeBroker`.
3. **`sciontool` Update**: Update `pkg/sciontool` to use the `SCION_HUB_TOKEN` from the environment if available.
4. **Middleware**: Add a `Bearer` token validation path in the Hub's authentication middleware that specifically handles these Hub-issued agent JWTs.

### Key Design Decisions

1. **HS256 (Symmetric) Signing**: Initial implementation uses HS256 for simplicity. RS256 can be configured for production deployments where asymmetric signing is preferred.

2. **Token in Custom Header**: Agents use `X-Scion-Agent-Token` header to avoid conflicts with user Bearer tokens. The middleware also accepts standard `Authorization: Bearer` for flexibility.

3. **Scope-Based Authorization**: The `scopes` claim enables fine-grained permissions. Default scope is `agent:status:update`.

4. **Environment Variables**: Tokens are passed to containers via:
   - `SCION_HUB_TOKEN`: The JWT token
   - `SCION_HUB_URL`: The Hub API endpoint
   - `SCION_AGENT_ID`: The agent's unique ID
