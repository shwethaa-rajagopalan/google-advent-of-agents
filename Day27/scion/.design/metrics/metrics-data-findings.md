# Research: Normalized Metrics and Logs for LLM Agents

## 1. Executive Summary
This document defines a normalized set of observability data—metrics, logs, and metadata—that is agnostic to specific LLM agent implementations (e.g., Gemini CLI, Codex, OpenCode). By standardizing these fields, we enable cross-platform analytics, consistent cost tracking, and unified performance monitoring across diverse "harnesses."

The schema is divided into **Universal** (ubiquitous across all agents) and **Common** (frequent and high-value but may vary by model or provider).

---

## 2. Logical Hierarchy
To normalize data across different architectures (OTel-native vs. JSON-log-based), we adopt a three-tier logical hierarchy:
1.  **Session (Trace):** The long-lived interaction (e.g., a CLI execution or a persistent chat thread).
2.  **Turn (Span/Step):** A single round-trip of User Input -> LLM Processing -> Tool Execution -> Final Response.
3.  **Event (Log/LogRecord):** Discrete occurrences within a turn (e.g., a specific tool call, an API error, or a token count update).

---

## 3. Normalized Metadata (Resource Attributes)
Every log or metric should be decorated with these attributes to allow for slicing and dicing.

| Attribute | Category | Description |
| :--- | :--- | :--- |
| `service.name` | Universal | The identity of the agent (e.g., `scion`, `codex`, `gcli`). |
| `session.id` | Universal | Unique UUID for the conversation/thread. |
| `user.id` | Universal | Anonymized or internal identifier of the user. |
| `model.name` | Universal | The specific model used (e.g., `gemini-1.5-pro`, `gpt-4o`). |
| `project.id` | Common | Hash or name of the repository/workspace being acted upon. |
| `env.terminal` | Common | User environment (e.g., `iterm2`, `vscode`, `tmux`). |
| `version` | Universal | The version of the agent software. |

## 4. Normalized Metrics
Metrics should follow OpenTelemetry semantic conventions for Generative AI where possible.

### 4.1 Usage & Cost (Counter/Histogram)
| Metric Name | Type | Description |
| :--- | :--- | :--- |
| `gen_ai.tokens.input` | Counter | Fresh tokens sent to the LLM. |
| `gen_ai.tokens.output` | Counter | Tokens generated by the LLM. |
| `gen_ai.tokens.cached` | Counter | Tokens read from the provider's cache (cost-saving). |
| `gen_ai.tokens.reasoning` | Common | Internal "thinking" tokens (e.g., OpenAI o1-series). |
| `gen_ai.cost.estimated` | Gauge | Calculated cost in USD based on token counts and model rates. |

### 4.2 Performance & Latency (Histogram)
| Metric Name | Type | Description |
| :--- | :--- | :--- |
| `gen_ai.latency.ttft` | Universal | Time to First Token (streaming latency). |
| `gen_ai.latency.total` | Universal | Total duration of the LLM generation. |
| `agent.tool.latency` | Universal | Time taken for tool/function execution. |
| `agent.turn.count` | Counter | Number of messages/steps per session. |

### 4.3 Reliability (Counter)
| Metric Name | Type | Description |
| :--- | :--- | :--- |
| `agent.tool.success` | Counter | Count of tool calls, tagged by `status=success/error`. |
| `gen_ai.error.count` | Counter | Count of API failures, tagged by `code` (e.g., 429, 500). |

---

## 5. Normalized Logs (Event Catalog)
Logs should capture structured data for debugging and auditing.

| Event Name | Attributes | Description |
| :--- | :--- | :--- |
| `agent.session.start` | `auth_mode`, `mcp_servers` | Emitted when a user initiates a chat. |
| `agent.user.prompt` | `prompt_length`, `content` (Redacted) | Capture of user input for intent analysis. |
| `agent.tool.call` | `tool_name`, `arguments` | Detailed record of an intent to use a tool. |
| `agent.tool.output` | `tool_name`, `status`, `exit_code` | Result of a tool execution. |
| `agent.approval.request` | `tool_name`, `decision` | User feedback on sensitive actions (e.g., `shell_execute`). |
| `gen_ai.response.done` | `tokens.*`, `finish_reason` | Final summary of an LLM completion. |

---

## 6. Implementation Strategies
*   **OTel-Native:** Directly instrument code using the OTel SDK, targeting `gen_ai.*` and `agent.*` namespaces.
*   **JSON-Log Aggregator:** For harnesses like Gemini CLI, a background process (or collector) should parse `session-*.json` files and transform them into the normalized OTel format described above.
*   **PII Masking:** All collectors/forwarders **must** redact or hash `user.email`, `agent.user.prompt`, and `agent.tool.output` (if it contains file content) unless explicit opt-in is provided.

---

## 7. Agnostic Language Mapping
To maintain a universal view of "what" the agent is working on, file extensions from tool calls should be mapped to normalized language names:
- `.ts`, `.tsx` -> `TypeScript`
- `.py` -> `Python`
- `.go` -> `Go`
- `.rs` -> `Rust`
- `.js`, `.jsx` -> `JavaScript`
- `.cpp`, `.cc`, `.hpp` -> `C++`
- `.rb` -> `Ruby`
- `.java` -> `Java`
- `.md` -> `Markdown`
- `.sh`, `.bash` -> `Shell`
