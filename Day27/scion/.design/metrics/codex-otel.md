# Codex OpenTelemetry Data Catalog

This document provides a technical research summary of the OpenTelemetry (OTel) data exported by Codex. This information is intended to inform the configuration of a custom OTel collector for filtering, sampling, and forwarding data.

## 1. Architecture Overview

Codex uses the standard OpenTelemetry SDK for Rust with the `tracing` ecosystem for instrumentation.

- **Exporters**: Supports OTLP via gRPC or HTTP (Binary/JSON).
- **Traces**: Standard OTel spans generated by `tracing-opentelemetry`.
- **Logs**: Structured events emitted via `tracing::event!` and bridged to OTel Logs via `opentelemetry-appender-tracing`.
- **Metrics**: Aggregated via the OTel Metrics SDK and exported using OTLP (Delta temporality).

## 2. Resource Attributes (Common Metadata)

All data (logs, traces, metrics) includes these resource-level attributes:

| Attribute | Description | Example |
| :--- | :--- | :--- |
| `service.name` | Identity of the service | `codex` |
| `service.version` | Binary version | `0.1.0` |
| `env` | Deployment environment | `prod`, `dev`, `staging` |

## 3. Log Events (Structured Events)

Codex uses an internal `OtelManager` to emit high-value structured events. These are exported as OTel Log records.

### Common Event Attributes
Most `codex.*` events include these fields:

- `conversation.id`: Unique Thread ID for the session.
- `auth_mode`: The auth provider used (`github`, `google`, etc.).
- `user.account_id`: Anonymized or internal account identifier.
- `user.email`: User email (if authenticated and available).
- `terminal.type`: User's terminal environment (e.g., `vscode`, `iterm2`).
- `model`: The AI model name (e.g., `gpt-4o`).
- `slug`: Specific deployment slug of the model.

### Event Catalog

| Event Name | Description | Key Attributes |
| :--- | :--- | :--- |
| `codex.conversation_starts` | Triggered when a new thread begins. | `provider_name`, `mcp_servers`, `sandbox_policy`, `approval_policy` |
| `codex.api_request` | LLM API call metrics. | `duration_ms`, `http.response.status_code`, `attempt`, `error.message` |
| `codex.sse_event` | Events from the AI stream. | `event.kind` (e.g., `text_delta`, `response.completed`), `duration_ms` |
| `codex.user_prompt` | Capture of user input. | `prompt_length`, `prompt` (Redacted unless `log_user_prompts` enabled) |
| `codex.tool_decision` | Result of tool use approval. | `tool_name`, `decision` (`approved`/`rejected`), `source` (`user`/`config`) |
| `codex.tool_result` | Outcome of a tool execution. | `tool_name`, `call_id`, `duration_ms`, `success`, `output` (preview) |

## 5. Traces

Spans are used to track latency and causality across the system.

- **`handle_responses`**: The root span for processing an AI response.
- **`tool_call`**: Spans tracking individual tool executions (e.g., file reads, shell commands).
- **Attributes**: Codex uses `otel.name` within spans to categorize logical steps (e.g., `function_call`, `text_delta`).

## 6. Collector Filtering Recommendations

To reduce volume and noise, a custom collector should consider:

1. **Log Filtering**: Forward only `event.name` matching `codex.*`. Filter out internal noise from third-party libraries (e.g., `reqwest`, `tokio`) unless at `ERROR` level.
2. **PII Masking**: Ensure `user.email` and `codex.user_prompt` are handled according to privacy requirements.
3. **Sampling**: High-frequency events like `codex.sse_event` (where `kind=text_delta`) can be sampled or aggregated before forwarding.
4. **Metric Aggregation**: Aggregate `codex.tool.call.duration_ms` at the collector level to reduce upstream traffic.
